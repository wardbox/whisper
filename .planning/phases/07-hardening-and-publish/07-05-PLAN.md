Create `.github/workflows/release.yml`:

```yaml
name: Release

on:
  push:
    branches: [main]

concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false

permissions:
  contents: write
  pull-requests: write
  id-token: write

jobs:
  release:
    name: Release
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 10

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'pnpm'
          registry-url: 'https://registry.npmjs.org'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Build library
        run: pnpm build

      - name: Create Release Pull Request or Publish
        uses: changesets/action@v1.7.0
        with:
          publish: pnpm release
          version: pnpm changeset version
          commit: 'ci: release'
          title: 'ci: version packages'
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
          NPM_CONFIG_PROVENANCE: 'true'
```

KEY POINTS:
- `id-token: write` is REQUIRED for provenance (Pitfall 7). Without it, provenance silently fails.
- `registry-url` on setup-node is REQUIRED for NODE_AUTH_TOKEN plumbing into ~/.npmrc.
- `pnpm release` runs `pnpm build && pnpm smoke && pnpm size && pnpm tarball:audit && pnpm changeset publish` (Plan 01).
- `changesets/action@v1.7.0` pinned explicitly (supply-chain guidance).
- `concurrency` prevents simultaneous release runs.
- No `workflow_dispatch` (D-14 — no manual escape hatch).
- `fetch-depth: 0` required for changesets git-log analysis.
- Gating via branch protection on main (manual setup in Task 4), not workflow_run.

Create `.github/workflows/token-check.yml`:

```yaml
name: NPM token check

on:
  schedule:
    - cron: '0 14 * * 1'
  workflow_dispatch:

permissions:
  contents: read

jobs:
  check:
    name: Validate NPM_TOKEN
    runs-on: ubuntu-latest
    if: github.repository == 'wardbox/whisper'
    steps:
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          registry-url: 'https://registry.npmjs.org'

      - name: Validate NPM_TOKEN via npm whoami
        env:
          NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}
        run: |
          set -e
          echo "Validating NPM_TOKEN..."
          if ! npm whoami 2>&1; then
            echo "::error::NPM_TOKEN is invalid or expired. Release workflow will fail."
            echo "::error::Rotate at npmjs.com > Access Tokens > Generate granular token (scoped to @wardbox/whisper)."
            echo "::error::Then update GitHub repo Settings > Secrets > Actions > NPM_TOKEN."
            exit 1
          fi
          echo "NPM_TOKEN is valid."

      - name: Rotation reminder
        run: |
          echo "::notice::Granular npm tokens expire every 90 days max. Rotate NPM_TOKEN every 80 days."
```

KEY POINTS:
- `npm whoami` validates the token is still active. 401 = expired or revoked.
- Weekly schedule (Mondays 14:00 UTC) balances signal vs noise.
- `workflow_dispatch` allows ad-hoc validation after rotation.
- `if: github.repository == 'wardbox/whisper'` prevents fork runs.
- `permissions: contents: read` only — zero write scope.

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions OIDC -> npm registry | Release workflow uses OIDC token for provenance attestation. If id-token: write is missing, provenance fails silently. |
| NPM_TOKEN secret -> npm registry | Granular token is the authentication for publish. If leaked, attacker can publish malicious versions scoped to @wardbox/whisper. |
| GitHub branch protection -> release workflow | Branch protection is the only gate preventing broken builds from being published. If not configured, merging to main triggers publish without CI. |
| Changesets config -> workspace packages | ignore field prevents docs from publishing. If misconfigured, Changesets versions and publishes the docs package. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-25 | Information Disclosure | NPM_TOKEN leaked via workflow logs | mitigate | GitHub Actions auto-masks secrets in logs; NPM_TOKEN is never echoed; token-check.yml uses NODE_AUTH_TOKEN (setup-node writes to .npmrc, not stdout) |
| T-07-26 | Elevation of Privilege | NPM_TOKEN has overly broad scope (manage settings, not just publish) | mitigate | Task 4 checkpoint explicitly instructs "Publish new versions ONLY" permission; D-13 locks this scope |
| T-07-27 | Tampering | Attacker merges to main without CI passing and triggers release | mitigate | Task 4 checkpoint configures branch protection requiring ci.yml status checks before merge to main |
| T-07-28 | Denial of Service | NPM_TOKEN silently expires after 90 days, breaking releases | mitigate | Task 3 token-check.yml runs weekly; Task 4 checkpoint instructs 80-day calendar reminder |
| T-07-29 | Spoofing | Published tarball not linked to verifiable commit (no provenance) | mitigate | release.yml sets id-token: write + NPM_CONFIG_PROVENANCE=true + publishConfig.provenance=true; Pitfall 7 documented |
| T-07-30 | Repudiation | Docs package accidentally published to npm as whisper-docs | mitigate | Task 1 sets ignore in Changesets config + private: true in docs package.json; two layers of defense |
| T-07-31 | Tampering | Supply-chain attack via floating action tag (changesets/action@v1) | mitigate | Task 2 pins to @v1.7.0 explicitly per RESEARCH supply-chain guidance |

</threat_model>
