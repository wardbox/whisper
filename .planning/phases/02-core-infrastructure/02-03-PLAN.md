<objective>
Build the proactive rate limiter -- the most complex and highest-risk component of whisper's core.

Purpose: The rate limiter prevents 429 errors by tracking Riot's multi-window rate limits proactively. It parses response headers to sync token buckets, queues requests when limits are near, and handles three distinct 429 types (app, method, service) with appropriate recovery strategies. This is the "magic where it makes sense" -- users get smart rate limiting without thinking about it.

Output: `rate-limiter.ts` and `rate-limiter.test.ts` in `packages/whisper/src/core/`.

**Internal types:**
```typescript
interface RateLimitBucket {
  limit: number;
  windowMs: number;
  remaining: number;
  resetAt: number;
}

interface ScopeState {
  buckets: RateLimitBucket[];
  queue: Array<{ resolve: () => void; reject: (err: Error) => void; timeoutId: ReturnType<typeof setTimeout> }>;
}
```

**RateLimiter class:**
```typescript
export class RateLimiter {
  private scopes: Map<string, ScopeState>;
  private config: Required<Pick<RateLimiterConfig, 'maxQueueSize' | 'requestTimeout'>> & RateLimiterConfig;

  constructor(config?: RateLimiterConfig | undefined);

  /** Acquire permission to make a request. Resolves when safe to proceed. */
  async acquire(route: string, methodId: string): Promise<void>;

  /** Update bucket state from response headers. Called after every successful response. */
  update(route: string, methodId: string, headers: Record<string, string>): void;

  /** Handle a 429 response. Classifies type and applies appropriate delay. */
  async handle429(route: string, methodId: string, headers: Record<string, string>, attempt: number): Promise<void>;
}
```

**Scope key construction:**
- App scope: `app:${route}` -- derived from `X-App-Rate-Limit` headers
- Method scope: `method:${route}:${methodId}` -- derived from `X-Method-Rate-Limit` headers

**acquire() logic:**
1. Build app scope key (`app:${route}`) and method scope key (`method:${route}:${methodId}`)
2. Check if both scopes exist in the map. If either is missing (cold start), allow through immediately.
3. If both scopes have buckets, check `canProceed(scope)` for BOTH. A request can only proceed if ALL buckets in ALL applicable scopes have `remaining > 0`.
4. If cannot proceed:
   - If `throwOnLimit` is true, throw `RateLimitError` immediately
   - If queue size >= `maxQueueSize`, throw `RateLimitError`
   - Otherwise, add to queue with a timeout. Set `setTimeout` for `requestTimeout` ms that rejects with `RateLimitError`.
   - Call `onRateLimit` callback if configured
   - Return a promise that resolves when a bucket resets and tokens are available

**update() logic:**
1. Parse `X-App-Rate-Limit` + `X-App-Rate-Limit-Count` headers into app scope buckets
2. Parse `X-Method-Rate-Limit` + `X-Method-Rate-Limit-Count` headers into method scope buckets
3. Store/replace buckets in the scope map
4. After updating, check if queued requests can now proceed (drain queue)

**handle429() logic:**
1. Call `classify429(headers)` to determine 429 type
2. For app/method: parse `Retry-After` header, wait that many seconds
3. For service: exponential backoff with jitter: `min(1000 * 2^attempt, 30000) + random * base * 0.1`

**canProceed() helper:** Returns true only if ALL buckets in scope have `remaining > 0`. Also handles expired buckets: if `Date.now() > bucket.resetAt`, reset `remaining` to `bucket.limit`.
